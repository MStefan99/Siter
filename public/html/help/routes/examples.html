To help you further understand how routes work, let's imagine the following scenario: you have two applications and you'd like to host them on a single server using Siter. The first application is a blog app and the second one is a file sharing app similar to Google Drive. In addition to these apps you'd also like to host public files from the file sharing app (let's call it a "Drive").

First you would add the route for your blog. The domain will be "blog" and port and prefix will use the default values. Set the target to server and use address according to your blog settings.

Then you would add a route for hosting files. Remember: more specific rules come first. To do this, create a route with domain "drive", prefix "public" and a default port. Make sure that the Drive does not also use this prefix. Set target to the directory containing public files of the Drive.

Finish by adding the route for the Drive app the same way as you did for the blog, but replace the domain with "drive". If needed, you can rearrange routes by dragging them.

Now your routes should look like this (note the order):

  1. blog.example.com:80/ for your blog
  2. drive.example.com:80/public/ for your public files
  3. drive.example.com:80/ for the Drive app

This way for your users the last two routes seem to be connected despite the fact that the second route is not even forwarded to the Drive app.

Now you just need to secure your routes and the setup is ready to use!
