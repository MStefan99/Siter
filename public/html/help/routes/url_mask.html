URL mask determines which route to choose for a request that Siter receives. The mask itself consists of three parts:

  • A subdomain
  • A port
  • And a prefix

To understand what they are for, let's take this URL as an example: http://service1.example.com:80/category1/page1/.

In this example, service1.example.com is a domain name of your website, the "name" by which your website is available on the Internet.

example.com is called a root domain. Currently Siter does not support route domain routing so this part is ignored when serving requests.

Our URL starts with a service1. This is a subdomain. Subdomains are a part of a domain name and are made to separate different services on your website.

The part after the domain is called a port and in our example it is 80. Different ports could be used to host multiple websites on a single server but usually they are set to a default value used by all browsers and a subdomain is used instead.

Our example URL ends in /category1/page1/. You can think of it as a file path that user requests. Here /category1/ acts like a directory which can have multiple files inside it, though they might not always be actual files on a server.

You can configure Siter to use a prefix for routing requests to a different target. For example, a request to website.com/shop/ could be routed differently than a request to just website.com.

Note that a prefix is removed when serving requests, so a request to website.com/shop/product1.html with /shop/ prefix will look for /product1.html on your server.

Be extremely careful when using route prefixes!

While route prefixes are a convenient way to separate routes, using them might have some unwanted consequences. Before adding a prefix you have to make sure your top-level website is not already using it (for example, there is no /shop/ section on website.com). Also if your target is not using links relative to a current directory (ones starting with "./") then such links might break when trying to host them on a route with prefix. In such cases you might want to use a subdomain instead.
